# GPU 实例故障排查手册

> 文档版本：v1.0
> 最后更新：2026-05-06
> 适用角色：客服、技术支持、运维
> 适用产品：GPU 云主机、AI 推理、AI 训练、容器 GPU 环境
> 标签：GPU 故障、CUDA、驱动、容器、训练、推理、排障

## 快速结论

- 先确认是不是买对了 GPU 规格，再看驱动、CUDA、容器和框架。
- `nvidia-smi` 报错时，先查驱动层，不要先改业务代码。
- 宿主机能识别 GPU，不代表容器里一定能识别。
- 训练慢不一定是 GPU 算力不够，也可能是 CPU、磁盘、数据加载或网络瓶颈。
- 排障时不要同时换镜像、驱动、CUDA 和框架版本，否则很难定位原因。

## 1. 适用范围

本文适用于以下问题场景：

- 创建 GPU 实例后，系统无法识别 GPU
- `nvidia-smi` 报错或不可用
- CUDA 不可用或框架识别不到 GPU
- 容器中无法使用 GPU
- 推理或训练性能异常
- 用户需要提单时，不知道该收集哪些信息

本文定位是排查与定位手册，不是镜像选型指南。环境选择原则请结合《GPU 实例镜像与驱动兼容指南》一起使用。

## 2. 故障分类总览

GPU 故障建议先分类，再排查。常见可分为五类：

1. 规格或资源选错
2. 驱动异常
3. CUDA 或运行时异常
4. 容器层异常
5. 性能异常

## 3. 第一层判断：是不是规格问题

### 3.1 是否真的购买了 GPU 实例

应先确认：

- 当前实例规格是否为 GPU 型
- GPU 数量是否符合预期
- GPU 型号与显存是否满足业务

### 3.2 是否存在显存不够的问题

很多程序跑不起来的根本原因不是 GPU 没有识别，而是模型放不下。

典型表现：

- 框架能识别 GPU
- `nvidia-smi` 正常
- 但模型加载时报显存不足或初始化失败

## 4. 第二层判断：驱动是否正常

### 4.1 用 `nvidia-smi` 做基础验证

建议首先检查：

- `nvidia-smi` 是否存在
- 能否正常执行
- 是否能显示 GPU 型号、驱动版本、显存信息

### 4.2 常见驱动异常表现

包括：

- 命令不存在
- 命令执行失败
- GPU 信息为空
- GPU 状态异常
- 驱动版本显示异常

### 4.3 常见驱动异常原因

- 使用了普通镜像
- 镜像环境不完整
- 驱动未安装成功
- 驱动与当前内核不兼容
- 手工升级驱动破坏了已验证环境
- 自定义镜像与新实例代际不兼容

## 5. 第三层判断：CUDA 是否正常

### 5.1 判断原则

驱动正常不代表 CUDA 一定正常。

建议继续确认：

- `nvcc --version`
- 框架是否报告 CUDA 可用
- 最小化示例是否能成功调用 GPU

### 5.2 典型异常表现

- CUDA 初始化失败
- 框架找不到 CUDA
- 运行时报版本不匹配
- 编译或加载扩展失败

### 5.3 常见原因

- 驱动版本过低
- CUDA 版本和驱动不匹配
- 框架所需 CUDA 版本与当前环境不同
- 环境变量或库路径冲突
- 宿主机与容器中的用户态运行时不一致

## 6. 第四层判断：框架和容器是否正常

### 6.1 容器场景的基本原则

如果业务跑在容器里，应先确认宿主机无误，再排查容器。

正确顺序应是：

1. 宿主机 GPU 是否正常
2. 宿主机驱动是否正常
3. 容器运行时是否支持 GPU
4. 容器内是否能看到 GPU
5. 容器内框架是否能使用 CUDA

### 6.2 容器中 GPU 不可用的常见原因

- 宿主机驱动已异常
- 容器未以 GPU 模式启动
- 容器镜像缺少必要用户态依赖
- 容器镜像和宿主机驱动组合不兼容

### 6.3 框架层异常

即使驱动和 CUDA 都正常，框架层仍可能有问题。

常见表现：

- PyTorch 只跑 CPU
- TensorFlow 不识别 GPU
- 推理框架加载模型时报错
- 自定义算子或扩展编译失败

## 7. 第五层判断：性能异常

很多 GPU 问题不是不能用，而是能用但很慢。

### 7.1 GPU 利用率低

典型表现：

- GPU 几乎空闲
- 训练速度慢
- 推理吞吐低
- CPU 占用反而偏高

常见原因：

- 数据读取慢
- batch size 太小
- CPU 不足
- 数据预处理过慢
- 存储吞吐不够
- 网络或分布式通信瓶颈

### 7.2 显存打满但吞吐不高

可能原因：

- 模型参数量大但算子效率不高
- 框架配置不合理
- IO 或数据预处理拖慢了整体速度
- 多卡配置不合理

### 7.3 多卡扩展性差

常见原因：

- 卡间通信效率低
- 节点间网络不足
- 数据切分不合理
- 分布式训练参数设置不当

## 8. 标准排查流程

建议所有 GPU 故障统一按以下顺序排查。

### 8.1 第一步：确认实例资源

确认：

- 实例是 GPU 型
- GPU 卡型和显存正确
- 地域和规格没有误选

### 8.2 第二步：确认宿主机驱动

确认：

- `nvidia-smi` 正常
- 驱动版本可见
- GPU 状态无明显异常

### 8.3 第三步：确认 CUDA

确认：

- CUDA 命令或运行环境可用
- 最小测试可执行
- 框架能看到 CUDA

### 8.4 第四步：确认容器或框架

确认：

- 容器运行参数正确
- 容器内能识别 GPU
- 框架版本和环境匹配

### 8.5 第五步：确认业务负载表现

确认：

- 是否显存不足
- 是否 CPU、磁盘或网络成为瓶颈
- 是否只是业务负载本身不适合当前配置

## 9. 场景化排查建议

### 9.1 刚创建实例就不工作

优先检查：

- 是否选错镜像
- 是否选错规格
- 官方推荐镜像是否可复现正常结果
- 是否实例本身存在底层异常

### 9.2 之前正常，升级后异常

优先检查：

- 是否升级过驱动
- 是否更换了 CUDA
- 是否升级了容器基础镜像
- 是否升级了框架主版本

### 9.3 容器中异常，宿主机正常

优先检查：

- 容器运行方式
- GPU 透传参数
- 容器镜像依赖
- 容器内 CUDA 用户态版本

### 9.4 推理能跑，训练报错

优先检查：

- 显存是否够
- 框架训练相关依赖是否完整
- 多卡或分布式依赖是否正确
- 数据读取链路是否稳定

## 10. 客服与运维的分工建议

### 10.1 客服可先确认的信息

客服可先引导用户提供：

- 实例规格
- 地域
- 操作系统版本
- 报错时间
- 报错截图
- 是否容器部署
- 是训练还是推理
- `nvidia-smi` 是否正常

### 10.2 需要转技术支持的情况

以下情况通常应升级处理：

- `nvidia-smi` 本身不可用
- 更换已验证镜像后仍不可用
- 多层环境同时异常无法快速判断
- 实例可能存在底层设备异常
- 用户已提供充分日志但仍无法定位

提工单时建议收集：

- 复现步骤

## 12. 常见误区

### 12.1 只要有 GPU 实例就一定能跑所有模型

错误。还需要显存、驱动、CUDA 和框架兼容。

### 12.2 `nvidia-smi` 正常就说明环境完全没问题

错误。驱动正常只是第一层，CUDA、框架和容器仍可能异常。

### 12.3 训练慢一定是 GPU 不够强

错误。很多时候真正瓶颈是数据加载、CPU、磁盘或网络。

### 12.4 升级到最新驱动一定更好

错误。生产环境更重要的是兼容性和稳定性，而不是版本最新。

## 13. 风险与注意事项

- 不建议在生产环境随意升级驱动
- 故障排查时不要同时替换多个组件
- 自定义镜像应有版本记录和验证记录
- 多卡训练和容器场景的复杂度明显高于单机单卡推理
- 处理性能问题时，应先确认是否为资源不足，再考虑环境异常

## 14. 对外回复建议

### 14.1 “为什么我的 GPU 用不了”

建议答法：

GPU 不可用通常需要分层排查，先确认实例规格是否正确，再确认驱动是否正常、CUDA 是否可用，最后再检查容器或框架环境。建议先执行 `nvidia-smi` 判断基础 GPU 状态。

### 14.2 “为什么我有 GPU 但模型还是跑不起来”

建议答法：

这不一定是 GPU 没有识别，也可能是显存不足、CUDA 与框架不兼容，或者容器中没有正确启用 GPU。需要结合显存需求、驱动状态和运行环境一起判断。

### 14.3 “为什么训练特别慢”

建议答法：

训练慢不一定是 GPU 算力不足，也可能是数据读取、CPU、磁盘或网络成为瓶颈。建议结合 GPU 利用率、显存占用和数据加载情况一起排查。

## 17. 标准 FAQ

### Q1：`nvidia-smi` 报错时最先该查什么

A：先确认实例是否真的是 GPU 规格，再确认镜像和驱动是否正常。`nvidia-smi` 本身不可用时，通常优先查驱动层，而不是先改框架代码。

### Q2：GPU 能识别但程序还是报错，说明什么

A：这通常说明驱动层已经基本正常，但可能仍存在显存不足、CUDA 不兼容、框架不匹配或容器配置异常等问题。

### Q3：训练慢是不是一定要换更大卡

A：不一定。训练慢也可能是数据读取、CPU、磁盘或网络成为瓶颈，建议结合 GPU 利用率和数据加载链路一起排查。

### Q4：为什么宿主机能用 GPU，容器里却不行

A：因为容器还依赖 GPU 透传参数、运行时配置和容器内依赖环境。宿主机正常只是前提，不代表容器一定正常。

### Q5：提单时为什么一定要附 `nvidia-smi` 输出

A：因为它能快速帮助判断 GPU 是否存在、驱动是否正常、显存和设备状态是否异常，是最有价值的基础诊断信息之一。
