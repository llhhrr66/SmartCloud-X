## 快速结论

- 安全组只开放必要端口，优先遵循最小权限原则。
- SSH 22 和 RDP 3389 不建议长期对 `0.0.0.0/0` 开放。
- 数据库、缓存、训练节点默认不要直接暴露公网。
- 端口不通时，按“实例状态 -> IP/路径 -> 安全组 -> 系统防火墙 -> 服务监听”顺序排查。
- 同一 VPC 不等于默认全互通，还要看安全组、路由和监听配置。

## 2. 核心概念

### 2.1 什么是 VPC

VPC 是逻辑隔离的专有网络环境，用于承载云上的计算、存储、数据库和其他资源。

可以把 VPC 理解为：

- 一块逻辑上独立的私有网络空间
- 资源之间进行内网通信的基础环境
- 安全组、子网、路由、网关等配置的承载层

### 2.2 什么是子网或交换机

子网用于在 VPC 内进一步划分网段和部署范围。通常用于：

- 按业务分区
- 按安全等级分区
- 按可用区分区

### 2.3 什么是安全组

安全组本质上是一组虚拟防火墙规则，用于控制资源的入站和出站流量。

可以把安全组理解为：

- 资源级别的访问控制边界
- 允许哪些来源访问哪些端口
- 限制哪些实例可与哪些实例互通

### 2.4 什么是入站规则和出站规则

#### 入站规则

控制谁可以访问这台实例。

#### 出站规则

控制这台实例可以访问谁。

## 3. 基本设计原则

### 3.1 最小权限原则

只开放必要端口，只允许必要来源。

### 3.2 分层访问原则

不同层的资源应采用不同访问策略。

常见分层：

- 公网入口层
- 应用服务层
- 数据层
- 运维管理层
- 内部训练或分析节点

### 3.3 管理端口与业务端口分离

管理端口通常包括：

- Linux SSH 22
- Windows RDP 3389

业务端口通常包括：

- HTTP 80
- HTTPS 443
- 应用服务端口

### 3.4 公网暴露最小化

以下资源应默认避免公网暴露：

- 数据库
- Redis 或缓存
- 消息队列
- 内部管理面板
- GPU 训练节点

### 3.5 云侧规则与系统侧规则配合

安全组放通，不等于服务一定可访问。

还受以下因素影响：

- 实例是否运行
- 是否有公网 IP 或正确出口
- 系统防火墙是否允许
- 应用进程是否在监听对应端口
- 路由是否正确

## 4. 常见端口与访问建议

### 4.1 Linux 管理端口

- SSH：22

建议：

- 不长期对 `0.0.0.0/0` 开放
- 优先限制为办公出口 IP、VPN 网段或堡垒机来源

### 4.2 Windows 管理端口

- RDP：3389

建议：

- 仅对受控来源开放
- 不建议长期开全网访问

### 4.3 Web 服务端口

- HTTP：80
- HTTPS：443

### 4.4 数据库类端口

例如：

- MySQL 3306
- PostgreSQL 5432
- Redis 6379

建议：

- 默认仅开放给应用层安全组或内网网段
- 不建议直接对公网开放

### 4.5 AI 或数据开发常见端口

例如：

- Jupyter 8888
- TensorBoard 6006

建议：

- 默认仅内网开放
- 如需公网访问，应增加身份认证和来源限制

## 5. 典型网络场景设计

### 5.1 单台网站服务器

建议配置：

- 80 和 443 对公网开放
- 22 或 3389 仅对固定管理来源开放
- 其他端口默认不开放

### 5.2 Web + 应用 + 数据库三层架构

建议配置：

- Web 层：开放 80 和 443
- 应用层：仅允许 Web 层访问其业务端口
- 数据库层：仅允许应用层访问数据库端口
- 管理端口只对运维来源开放

### 5.3 仅内网使用的业务系统

建议配置：

- 不分配公网访问能力，或不开放公网端口
- 业务端口仅对内网网段开放
- 管理端口只对更小范围来源开放

### 5.4 GPU 推理节点

建议配置：

- 推理端口仅开放给上游应用层
- 管理端口仅对运维来源开放

### 5.5 GPU 训练节点

建议配置：

- 默认内网访问
- 不直接开放公众访问端口
- 若开放 Jupyter 或调试端口，必须限制来源

## 6. 标准配置步骤

### 6.1 第一步：先明确资源角色

在配安全组前，先给实例定角色：

- 公网入口
- 业务服务
- 数据库
- 运维节点
- GPU 推理
- GPU 训练

### 6.2 第二步：列出实际需要开放的端口

明确：

- 哪个端口用于什么服务
- 谁需要访问它
- 是公网来源还是内网来源
- 是否需要长期开放

### 6.3 第三步：先配最小可用规则

建议先配置最小集合：

- 必需业务端口
- 必需管理端口
- 必需内网互访规则

### 6.4 第四步：上线前做可达性校验

至少确认：

- 实例状态正常
- 公网或内网 IP 正确
- 安全组规则已生效
- 系统防火墙未拦截
- 应用进程确实监听端口

## 7. 端口不通排查流程

### 7.1 先确认实例状态

检查：

- 实例是否启动
- 系统是否健康
- 是否有异常重启或系统故障

### 7.2 再确认网络入口条件

检查：

- 是否有公网 IP 或正确内网路径
- 访问方是否在允许的网段内
- DNS 是否解析到正确地址

### 7.3 再确认安全组规则

检查：

- 目标端口是否开放
- 协议是否正确
- 来源网段是否包含当前访问方

### 7.4 再确认系统防火墙和服务监听

检查：

- 操作系统防火墙是否放行端口
- 应用是否已启动
- 应用是否监听在正确地址和端口

## 8. 实例无法登录的排查要点

### 8.1 Linux SSH 登录失败

优先检查：

- 22 端口是否开放
- 来源 IP 是否被允许
- 密钥或密码是否正确
- 用户名是否正确
- 实例是否分配了正确公网地址
- SSH 服务是否正常运行

### 8.2 Windows RDP 登录失败

优先检查：

- 3389 是否开放
- 账号密码是否正确
- Windows 防火墙是否阻断
- 系统是否卡在启动异常状态

### 8.3 管理端口临时开放的风险

如果为了排障临时扩大来源范围，应在排障完成后尽快收回。

## 9. 常见误区

### 9.1 安全组开了，服务就一定能通

错误。安全组只是云侧入口条件之一。

### 9.2 只要是内网就一定安全

错误。内网也需要按来源控制访问，尤其数据库和管理端口。

### 9.3 数据库临时开放公网没关系

高风险。很多安全事故就是从临时开放忘记关开始的。

### 9.4 调试方便先把所有端口全开

短期看省事，长期看最危险，也会让后续排查变得更复杂。

## 10. 风险与注意事项

- 管理端口长期全网开放风险极高
- 业务上线应有最小开放面原则
- 数据类资源默认不应开放公网
- 安全组变更应有记录、审批或至少留痕

## 11. 对外回复建议

### 11.1 “为什么我端口开了还是访问不到”

建议答法：

端口是否可访问，不仅取决于安全组，还取决于实例状态、公网或内网地址、系统防火墙以及应用是否真正监听该端口。建议按实例状态、网络地址、安全组、系统防火墙、服务监听顺序逐步排查。

### 11.2 “数据库能不能直接开公网给我连”

建议答法：

除非有明确且受控的业务需要，否则不建议直接开放数据库公网访问。更推荐通过应用层、堡垒机、VPN 或受限来源访问，降低暴露风险。

### 11.3 “SSH 能不能直接对全网开放”

建议答法：

不建议长期对全网开放。更安全的做法是限制为办公出口 IP、VPN 或堡垒机来源。

## 14. 标准 FAQ

### Q1：为什么安全组已经开了，服务还是访问不到

A：因为安全组只是云侧入口条件之一。实例状态、公网 IP、系统防火墙、应用监听地址和服务本身状态，都会影响最终能否访问。

### Q2：数据库能不能临时开公网

A：除非有明确且可控的短期需求，否则不建议。更稳妥的方式是通过内网、堡垒机、VPN 或应用层访问，避免数据库长期暴露公网。

### Q3：SSH 和 RDP 为什么不建议对全网开放

A：因为管理端口是高风险入口。长期对全网开放会显著增加暴力破解、误暴露和横向渗透风险。

### Q4：为什么同一个 VPC 里资源还会互相访问不到

A：同一 VPC 只说明网络在同一私有环境中，不代表默认全部互通。还要检查安全组、路由、网段和服务监听配置。

### Q5：Jupyter 或调试端口能不能直接放公网

A：不建议直接开放。即使短期调试，也应限制来源并结合认证措施，避免敏感数据和计算资源被滥用。
