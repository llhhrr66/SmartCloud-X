# 客服/运营/技术支持工单提报模板

> 文档版本：v1.0
> 最后更新：2026-05-06
> 适用角色：客服、运营、技术支持、财务支持、合规支持
> 适用产品：工单系统、问题升级、跨团队协作
> 标签：工单、提单、升级、交接、优先级、排障

## 快速结论

- 好工单的核心不是写“有问题”，而是写清资源、时间、影响、已排查内容和期望结果。
- 高优先级工单最怕信息缺失，因为会直接拖慢响应和升级判断。
- 咨询类、故障类、财务类、备案类、活动类工单应分开处理，不要混在一起。
- 没有截图、日志、时间点和资源标识，接手团队通常无法高效定位。
- 工单标题最好带上问题类型、优先级和资源标识，方便快速分拣。

## 高频问法入口

- 工单到底要写什么
- 为什么还要补这么多信息
- P1 和 P2 怎么区分
- 技术工单和咨询单有什么不同
- 为什么一定要写已做排查

## 1. 适用范围

本文适用于以下场景：

- 客服无法直接解决问题，需要转交技术、财务、运营或合规团队
- 运营活动出现异常，需要提工单给技术或数据团队
- 用户反馈账单、备案、登录、GPU、网络等问题，需要进入正式协作流程
- 需要规范工单字段、问题分级、升级路径和交接方式

## 2. 为什么工单模板重要

一个好的工单，要让接手人迅速回答以下问题：

- 用户是谁
- 哪个资源有问题
- 问题是什么
- 什么时候开始的
- 影响范围有多大
- 已经排查过什么
- 期望怎么解决
- 是否紧急

## 3. 工单分类标准

建议至少分为以下几类：

### 3.1 产品咨询类

适用场景：

- 用户不确定买什么规格
- 想咨询地域、带宽、GPU、备案要求等

### 3.2 登录与网络类

适用场景：

- SSH 或 RDP 无法登录
- 端口不通
- 公网访问失败
- 内网互通失败

### 3.3 GPU 与技术故障类

适用场景：

- GPU 不识别
- CUDA 异常
- 容器无法使用 GPU
- 训练或推理性能异常

### 3.4 财务与账单类

适用场景：

- 异常扣费
- 发票问题
- 退款或退订问题
- 账单解释需求

### 3.5 ICP 与合规类

适用场景：

- 备案被驳回
- 材料不明确
- 主体、域名、接入信息不一致

### 3.6 营销与活动类

适用场景：

- 活动链接异常
- 二维码失效
- 渠道统计异常
- 活动规则展示不一致

## 4. 工单优先级建议

### 4.1 P1 高优先级

适用场景：

- 核心业务中断
- 大面积登录失败
- 生产系统无法访问
- 高价值客户重大故障
- 正在进行中的关键活动大面积异常

### 4.2 P2 中优先级

适用场景：

- 单客户严重问题
- 某类功能故障
- 高影响但未全局中断的问题

### 4.3 P3 普通优先级

适用场景：

- 咨询类问题
- 单点非紧急问题
- 可排期处理的问题

### 4.4 P4 低优先级

适用场景：

- 经验性建议
- 优化类需求
- 非当期必须处理的问题

## 5. 工单必填字段模板

### 5.1 基础信息

- 工单标题
- 工单类别
- 优先级
- 提单时间
- 提单人
- 所属团队

### 5.2 用户信息

- 用户 ID
- 用户名称
- 联系方式
- 租户 ID（如适用）
- 用户类型（个人、企业、内部）

### 5.3 资源信息

根据问题类型填写，例如：

- 实例 ID
- 地域和可用区
- 域名
- 订单号
- 发票号
- 活动 ID
- 推广链接 ID
- 知识库文档 ID
- GPU 规格
- 云盘 ID

### 5.4 问题描述

建议用固定结构写：

- 发生了什么问题
- 从什么时候开始
- 用户执行了什么操作
- 实际结果是什么
- 期望结果是什么

### 5.5 影响范围

至少说明：

- 影响单个用户还是多个用户
- 影响单个资源还是整类资源
- 是否影响生产
- 是否影响交易、下单、登录、训练或访问

### 5.6 已做排查

至少说明：

- 已检查了什么
- 已确认什么正常
- 已尝试什么修复
- 当前怀疑方向是什么

### 5.7 附件与证据

建议附上：

- 报错截图
- 日志片段
- `nvidia-smi` 输出
- 控制台截图
- 账单截图
- 域名状态截图
- 推广链接截图
- 复现录屏

## 6. 标准标题模板

建议工单标题可采用以下结构：

`[问题类型][优先级][资源标识] 问题摘要`

例如：

- `[登录故障][P1][i-123456] Linux实例SSH无法连接`
- `[GPU异常][P2][i-abcdef] 容器内无法识别GPU`
- `[账单异常][P2][order-202605] 用户反馈重复扣费`
- `[备案退回][P3][example.com] 域名实名信息与主体不一致`
- `[营销活动][P2][campaign-gpu-q2] 二维码跳转错误页面`

## 7. 各类工单示例模板

### 7.1 登录故障类工单示例

建议包含：

- 实例 ID
- 操作系统
- 登录方式
- 使用的 IP
- 报错内容
- 是否检查过安全组
- 是否近期变更过密码、密钥、镜像、网络

### 7.2 GPU 故障类工单示例

建议包含：

- 实例规格
- GPU 型号
- 驱动版本
- CUDA 版本
- 是否容器部署
- 训练还是推理
- `nvidia-smi` 输出
- 框架版本
- 报错日志

### 7.3 财务账单类工单示例

建议包含：

- 用户 ID
- 订单号
- 账期
- 金额
- 问题描述
- 是否核查自动续费
- 是否核查快照、云盘、公网资源费用

### 7.4 ICP 备案类工单示例

建议包含：

- 主体类型
- 域名
- 驳回原因截图
- 已提交材料清单
- 域名实名状态
- 主体与域名是否一致
- 当前希望处理的问题

### 7.5 营销活动类工单示例

建议包含：

- 活动 ID 或名称
- 渠道名称
- 推广链接
- 二维码版本
- 异常开始时间
- 受影响用户范围
- 实际跳转页面
- 预期跳转页面

## 8. 升级规则建议

### 8.1 什么情况需要立即升级

建议立即升级的情况包括：

- 生产业务中断
- 多客户同时受影响
- 登录、支付、下单、GPU 训练等核心流程中断
- 正在进行中的营销活动主入口异常
- 备案、财务类高价值客户时效性问题

### 8.2 什么情况可先补信息再流转

例如：

- 普通咨询
- 单用户非紧急报障
- 信息明显缺失的工单

## 9. 交接规范

### 9.1 客服转技术

应重点说明：

- 用户影响
- 是否复现
- 已做基础排查
- 希望技术确认的具体问题

### 9.2 运营转技术

应重点说明：

- 活动名称和时间
- 渠道和链接
- 影响范围
- 是否正在投放中
- 是否影响注册或下单转化

### 9.3 客服转财务

应重点说明：

- 订单号
- 账期
- 金额
- 用户质疑点
- 已核对的基础计费信息

### 9.4 客服转备案或合规

应重点说明：

- 主体类型
- 域名
- 驳回原因
- 已提交材料
- 用户当前诉求

## 10. 常见问题

### 10.1 为什么不能只写“用户说不能用”

因为这类描述没有资源标识、没有复现路径、没有影响范围，也没有已排查信息，接手团队无法高效处理。

### 10.2 为什么已做排查必须写

因为很多问题不是没人会查，而是每个人都在重复查同一轮基础信息。

### 10.3 为什么工单要分优先级

因为不是所有问题都要同样的响应速度。统一优先级有助于资源调度、响应承诺和升级决策。

## 11. 风险与注意事项

- 不完整工单会显著拖慢处理速度
- 高优先级工单必须避免信息缺失
- 工单标题不要模糊
- 日志、截图和时间点越明确，定位越快
- 不要把咨询和故障混成同一类处理流程

## 12. 对外回复建议

### 12.1 “为什么还要我补这么多信息”

建议答法：

补充资源标识、报错时间、截图和操作路径，能帮助技术或财务团队更快定位问题，减少来回沟通时间，也能避免重复排查。

### 12.2 “我已经说了有问题，为什么还要工单”

建议答法：

工单的作用是把问题转成可追踪、可升级、可复现的记录，便于不同团队接手和持续跟进，尤其在涉及技术、财务、备案或活动异常时更重要。

## 15. 标准 FAQ

### Q1：为什么不能只写一句“用户说不能用”

A：因为这样的工单没有资源标识、没有时间点、没有报错信息、没有影响范围，接手团队无法高效定位问题，只会反复补资料。

### Q2：为什么工单里一定要写“已做排查”

A：因为这能避免不同团队重复做同一轮基础检查，提高流转效率，也有助于更快判断问题该转给谁。

### Q3：什么问题算 P1

A：通常是核心业务中断、大面积受影响、生产不可用或正在进行中的关键活动主入口异常，这类问题需要立即响应。

### Q4：咨询问题和故障问题为什么要分开提

A：因为两者处理路径不同。咨询类更偏方案建议，故障类更偏定位和恢复，如果混在一起会影响优先级和流转效率。

### Q5：为什么附件和截图这么重要

A：因为很多技术、账单、备案或活动问题都依赖上下文和证据。没有截图、日志或时间点，问题通常难以复现和确认。
