# OpenAPI Placeholders

Foundation owns this directory as the landing zone for shared and service-level API specifications.

Current baseline now includes:
- `components.openapi.yaml` for shared headers, response headers, reusable responses, shared internal schemas, canonical external envelope schemas, pagination parameters, and user-facing task DTO refs
- current route baselines for orchestrator, tool-hub, and business-tools services, including canonical paths plus documented legacy/internal/MCP aliases where they exist, and the current `/healthz` plus additive `/readyz` readiness/runtime-health semantics for the infra-migration path
- an external `admin-api.openapi.yaml` placeholder baseline for admin dashboard, knowledge/document management, document-detail/job polling, and retrieval diagnostics routes
- the current orchestrator session-management surface for session create/list/detail/update, history reads, retry, archive/restore, and `/api/v1/chat/completions`, including dependency-aware tool-plan/task/handoff metadata for session-context hydration planning
- the current orchestrator continuation/cancellation surface for `POST /api/v1/chat/sessions/{conversation_id}/continue` and `POST /api/v1/chat/sessions/{conversation_id}/cancel`
- the current orchestrator soft-delete baseline for `DELETE /api/v1/chat/sessions/{conversation_id}` plus the additive chat-completion compatibility/summary fields exposed by the live orchestrator implementation
- the current orchestrator SSE route placeholder, including the live emitted event sequence, promoted stream payload schemas, and stored message-event replay/list routes plus the shared `CHAT_STREAM_EVENTS_NOT_FOUND` replay-miss baseline
- the current tool-hub audit read placeholders plus tool preflight/clarification coverage for non-audited readiness checks, richer registry metadata (`version`, `input_schema`, `output_schema`, `session_context_bindings`, `session_context_output_keys`, `prerequisite_tool_names`), direct tool-call fidelity fields (`status`, `summary`, `result`, `citations`) with compatibility `message`/`data` aliases, additive audit `summary`/`citations`, surfaced preflight `session_context_bindings` plus execution-policy metadata (`tool_mode`, `timeout_ms`, `idempotent`, `cache_ttl_seconds`), compensation-call bridge placeholders, `audit_tag` filtering, cache-replay annotations such as `cache-hit`, malformed-payload `invalid-payload` audit visibility, and rollback flows
- the current business-tools provider-discovery routes for internal descriptor lookup and readiness preflight when tool-hub is configured for HTTP transport
- placeholder-ready specs for RAG and knowledge services with the current known route surfaces reserved for downstream owners, including knowledge import preview / file-import routes, the live knowledge runtime snapshot and internal admin-audit inspection routes, plus documented temporary gateway-backed admin retrieval/search routes
- a new `auth-user-service.openapi.yaml` placeholder for end-user login/account flows, current auth compatibility aliases (`/auth/profile`, `/auth/change-password`, `/auth/forgot-password`, `/auth/reset-password`), admin auth bootstrap, internal permission-check/cache-invalidation contracts, and runtime-health guidance notes for future backend-evidence publication
- user-facing placeholder specs for marketing-service and research-service, including canonical external envelopes plus frozen campaign/copy/promotion-link/poster and research-task routes, additive task status/result helper routes, persisted marketing artifact history/detail reads, reusable result/list DTO placeholders, and runtime-health guidance notes for future backend-evidence publication

These files are still draft contracts, but they are no longer empty stubs. Downstream supervisors should extend them through the change-request workflow instead of maintaining parallel contract definitions elsewhere.
