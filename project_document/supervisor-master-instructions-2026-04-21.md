# SmartCloud-X 三路 Hermes Supervisor 总控说明

## 目标
启动 3 个长期运行的 Hermes supervisor，各自严格在自己的文档与代码边界内，执行以下闭环：

1. 阅读负责范围内的拆分开发文档与现有代码
2. 生成并持续维护自己的《开发需求确认与跟踪文档》
3. 按文档要求和代码现状差异进行开发完善
4. 主动补测试、跑验证
5. 自审代码，发现风险并修复
6. 每完成一项就更新表格
7. 持续循环，直到本边界内表格项全部完成或仅剩跨边界/阻塞项

## 核心原则
- 先读文档和代码，再写开发需求确认文档，再开发。
- 不允许脱离表格乱改代码。
- 每次只推进一个当前项，做完后必须测试、review、更新文档。
- 只允许修改自己边界内的目录。
- 发现跨边界问题，只能记录为 `cross_boundary` 或 `blocked`，不得直接改别组目录。
- 不允许通过删除测试、跳过验证、弱化契约来换取“完成”。
- 不允许仅根据 README、状态文档、已有 md 说明就判定“已完成”或“已对齐”。
- 完成判断必须同时基于：真实代码实现、开发文档规范、自己亲自运行的真实测试/验证结果。
- 若 README / md 声称已实现，但代码、测试、运行结果不能证实，则必须按未完成或有风险处理。

## 八荣八耻（必须遵守）
- 以瞎猜接口为耻，以认真查询为荣。
- 以创造接口为耻，以复用现有为荣。
- 以跳过验证为耻，以主动测试为荣。
- 以破坏架构为耻，以遵循规范为荣。
- 以假装理解为耻，以诚实无知为荣。
- 以盲目修改为耻，以谨慎重构为荣。

## 通用执行循环
### 第 1 阶段：建立并初始化跟踪文档
每个 supervisor 启动后必须先创建或更新自己的：
- `docs/status/<supervisor-name>-dev-review.md`

必须先完成：
1. 阅读 `00-开发文档总索引.md`
2. 阅读自己负责的拆分文档
3. 阅读自己边界内的 README、主代码、测试、相关 OpenAPI/配置
4. 产出开发需求确认与跟踪表

### 第 2 阶段：按表逐项执行
每一轮循环必须：
1. 从表格中选择一个 `pending` / `review_required` / `testing` 项
2. 开发或修复该项
3. 运行对应验证
4. 自审变更是否符合文档、是否越界、是否引入回归
5. 更新表格状态和结果
6. 再进入下一项

### 第 3 阶段：持续收敛
当本边界内只剩：
- `cross_boundary`
- `blocked`
时，才允许宣布本边界已收敛。

## 状态字段约束
表格状态只允许使用：
- `pending`
- `in_progress`
- `review_required`
- `testing`
- `completed`
- `blocked`
- `cross_boundary`

## 越界规则
### 允许
- 修改自己负责目录下的实现、测试、README、必要的局部文档
- 修改自己的跟踪文档

### 只读允许
- 阅读总索引、相邻拆分文档、其他服务代码、OpenAPI、共享目录

### 禁止
- 修改其他 supervisor 负责的 app 目录
- 修改未授权共享包以规避本问题
- 修改别组测试来让自己通过
- 破坏既有契约而不记录为跨边界风险

## 停止条件
### 可以停止
仅在以下条件满足时才允许停止：
1. 本边界内所有可落地项都已 `completed`
2. 剩余仅为 `cross_boundary` 或 `blocked`
3. 已在文档中写清验证结果、残留风险、未处理原因

### 必须停止并报告 BLOCKER
命中任一情况立即停止当前轮并输出 BLOCKER：
1. 需要修改其他 supervisor 边界内的代码才能继续
2. 需要修改冻结契约 / OpenAPI / 权限码 / 错误码 / 表结构且不在本边界授权内
3. 同一问题已尝试超过 3 次仍未解决
4. 两轮修复后验证仍然失败
5. 发现高风险安全/权限/数据一致性问题但无法在本边界内安全修复

## 输出口径
只允许使用清晰口径：
- 本轮完成 / 本轮未完成
- 已完成项 / 未完成项 / 阻塞项 / 跨边界项

禁止模糊表述：
- 差不多
- 基本完成
- 部分接入
- 半完成
- 应该可以
